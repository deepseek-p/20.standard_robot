# AI 交接规则（必读）

适用范围：本仓库（`20.standard_robot`）内所有 AI 协作开发活动。  
目标：保证嵌入式改动可追踪、可验证、可回滚，避免“能跑但不可维护”。

## 0. 最小执行流程

1. 先读：`docs/01_PROJECT_OVERVIEW.md`、`docs/03_TASK_MAP.md`、`docs/04_MODULE_MAP.md`。
2. 明确改动边界：先确认是否会触发“必须写 ai_sessions”。
3. 实施改动并记录验证证据。
4. 按模板产出会话记录：`docs/ai_sessions/YYYY-MM-DD_模块_行为.md`。
5. 若存在未验证前提，必须单独写 `⚠ 未验证假设` 小节。

## 1. 什么时候必须写 ai_sessions

满足任一条即必须写：

- 修改 `Src/freertos.c`：任务创建、优先级、栈大小、创建顺序。
- 修改中断与 HAL 回调路径：如 CAN/UART/EXTI/DMA 回调（含 `HAL_CAN_RxFifo0MsgPendingCallback`、`USARTx_IRQHandler`、`HAL_GPIO_EXTI_Callback`、DMA IRQ）。
- 修改 `application/CAN_receive.c`：CAN 接收解析、发送电流、ID/槽位映射。
- 修改 `application/gimbal_behaviour.c`：云台行为状态机或控制模式切换逻辑。
- 修改 `application/chassis_power_control.c`：功率限制策略、限流缩放逻辑。
- 修改 `application/referee*.c`：裁判系统协议解析、功率/热量数据依赖。
- 修改 `application/remote_control.c`：SBUS 解析、DMA 双缓冲、掉线恢复。
- 修改 `application/INS_task.c` 或 `components/algorithm/AHRS*`：姿态解算链路、IMU 中断/DMA 时序。
- 修改 `components/controller/pid.*` 或关键 PID 参数宏：控制环稳定性会受影响。
- 任何影响实时性、上位机通信、功率控制、发弹安全、失联保护的改动。

## 2. 每次记录必须包含的字段

以下字段缺一不可：

- 目标（这次要解决什么）
- 改动文件列表（完整路径）
- 关键改动摘要（按文件说明）
- 关键决策与理由（为何这样做，不选什么）
- 实时性影响（阻塞/CPU/栈/帧率）
- 风险点（功能风险、时序风险、硬件风险）
- `⚠ 未验证假设`（必须单独成节）
- 验证方式（至少覆盖：串口/USB/示波器/上位机/长跑中的适用项）
- 回滚方式（如何恢复到改动前）

## 3. 嵌入式专项检查清单（提交前必查）

- [ ] 是否修改了 `/* USER CODE BEGIN */` 与 `/* USER CODE END */` 之外的 CubeMX 自动生成区域。
- [ ] 是否重新核对 `Inc/FreeRTOSConfig.h` 中 `configTICK_RATE_HZ = 1000`（`1 tick = 1ms`）对周期计算的影响。
- [ ] 是否改变了 CAN 电流槽位映射（`0x201~0x204` 底盘、`0x205~0x207` 云台/拨弹）。
- [ ] 是否引入了 `osDelay/vTaskDelay` 之外的阻塞路径（忙等、长时间轮询、阻塞式串口/总线访问）。
- [ ] 是否需要并已观测 `uxTaskGetStackHighWaterMark`（尤其 `calibrate/chassis/detect/gimbal`）。
- [ ] 是否新增了跨中断与任务共享变量，且已评估竞态条件。
- [ ] 是否给出可执行的板级验证步骤（不是“理论可行”）。

## 4. 会话文件命名规范

- 目录：`docs/ai_sessions/`
- 文件名：`YYYY-MM-DD_模块_行为.md`
- 示例：`2026-02-19_chassis_power_limit_tune.md`

## 5. ⚠ 未验证假设（强制规则）

每篇会话记录都必须包含独立章节标题：`## ⚠ 未验证假设`。

该章节至少写明：

- 假设内容（具体到宏/函数/任务/外设）
- 为什么当前无法验证（缺设备/缺工况/缺数据）
- 对系统的潜在影响
- 后续验证计划（负责人、方法、触发条件）

未写该节，视为交接不完整。

## 6. 禁止盲改——根因定位前置规则

背景：pitch 下冲问题从 2026-02-20 到 2026-02-22 连续 4 轮修改均未解决，根本原因是每次都在未确认故障链路的情况下就添加防御代码，导致宏和分支堆积但问题依旧。

### 6.1 改代码前必须先回答三个问题

在对控制环路（PID、限幅、模式切换）做任何修改之前，AI 必须在会话记录中明确写出：

1. **当前故障的实际执行路径是什么？** —— 用实机数据（USB/串口日志）证明故障发生时系统运行在哪个模式、哪个函数分支，不允许靠代码静态阅读猜测。
2. **本次修改命中的是哪条路径？** —— 必须与第 1 点对应，若改动路径与故障路径不一致，禁止提交。
3. **如何用数据证明本次修改生效？** —— 给出具体的观测列/变量和预期变化方向，不接受"待验证"。

缺少以上任一项，视为改动不合规，不得进入代码实施阶段。

### 6.2 同一问题连续修改不得超过 2 轮未闭环

- 若针对同一现象已连续做了 2 次代码修改且均未拿到"实机数据证明改善"的闭环证据，**第 3 次必须停下来**，不得继续叠加新的防御代码。
- 此时 AI 应：
  - 汇总前几轮改动与数据，列出已排除和未排除的假设。
  - 向用户提出针对性的数据采集方案（指定变量、场景、采样时机），而不是继续猜测并改代码。
  - 在 `risk_log.md` 中将该问题标记为 `Blocked - 需数据`。

### 6.3 禁止防御性宏无限堆积

- 针对同一模块/同一现象，编译期开关宏不得超过 3 个。超出时必须先评估已有宏是否有效，无效的应清理后再新增。
- 每个新增宏必须在会话记录中注明：**该宏解决的是哪条已确认的故障路径**，不允许"以防万一"式添加。

### 6.4 改动前必须确认当前运行模式

- 涉及多模式系统（如 gimbal 的 GYRO/ENCONDE 模式、chassis 的 FOLLOW/NO_FOLLOW 模式）时，必须先通过实机日志确认故障工况下的实际模式值。
- 会话记录中必须引用具体数据帧（如 `pitch_mode=1` 表示绝对角模式），不接受"应该是 XX 模式"的推测。

## 7. VOFA+ 验收回填（精简）

### 7.1 默认验收来源

- `data.md` 和“下一轮口述验收结果”都有效。
- 若两者冲突，以口述结论为主；同时在会话记录中注明冲突点。

### 7.2 本轮先做回填

- 上一轮只要改了控制链路（模式/PID/限幅/坐标变换），本轮先补“上轮验收结论”，再继续改码。

### 7.3 无验收结果时的规定

- 若没有新 VOFA+ 或明确口述结论：
  - 在 `risk_log.md` 标记 `Blocked - 需数据`。
  - 写清需要补的数据列与场景。
  - 除非用户明确要求或属于安全止损，不继续叠加新防御改动。

### 7.4 会话记录最小字段

- 每次会话必须有“上轮验收结论”，至少包含：
  - 上轮会话
  - 数据来源
  - 关键帧/时间段
  - 状态（`Passed/Partial/Failed/Blocked`）
  - 差异
  - 下一步

### 7.5 状态映射统一

- `Passed -> Mitigated（待长跑）`
- `Partial -> In Progress`
- `Failed -> Open`
- `Blocked -> Blocked - 需数据`

### 7.6 最小文档同步集

- 每次回填至少同步 3 处：
  - 对应 `ai_sessions`
  - `risk_log.md`
  - `ai_sessions/CURRENT_STATE.md`
- 若新建会话文件，再同步 `docs/INDEX.md`。
